# MyChordpro 同期方針 基本設計書

## 1. 適用範囲

本書は、`MyChordpro` を `chordwiki_personal` に寄せる実装の基本設計を定義する。  
以下を強制要件とする。

- セットリスト共有機能は実装しない
- `MyChordpro` の認証方式（GitHub 認証）を維持する
- `ownerId` モデルを維持し、他システムと統合しない
- URL 共有で他者に同一データを見せる仕様にしない

## 2. 設計方針

### 2.1 設計原則

- 機能同等性よりも運用境界（認証・データ分離）を優先する
- UI 同期は「見た目」だけでなく「状態遷移と保存ルール」まで揃える
- API は後方互換を保ち、破壊的変更を避ける

### 2.2 境界条件

- 認証境界: GitHub principal を唯一のユーザー識別とする
- データ境界: 全曲データは `ownerId` で完全分離する
- 共有境界: 共有URL、公開リンク、共同編集を提供しない

## 3. 画面基本設計

### 3.1 トップページ（`/`）

#### 表示要素（トップ）

- 検索フォーム（`target=song|tag`）
- タグサジェスト
- ランキング/検索結果一覧
- ページネーション
- 新規追加導線（認証済みのみ）

#### 挙動（トップ）

- `q` 未指定時はランキング表示
- `q` 指定時は検索表示
- URLクエリ（`q`, `target`, `page`）と画面状態を同期
- `/?page=1` は `/` に正規化

#### 非対応（トップ）

- ユーザー横断検索
- 公開曲の探索

### 3.2 曲詳細ページ（`/song.html`）

#### 表示要素（曲詳細）

- ChordPro 描画領域
- Song Controls（移調/表記、メモ/手書き、オートスクロール、表示設定）
- Tags/YouTube パネル
- YouTube ミニプレイヤー

#### 挙動（曲詳細）

- 曲表示時に閲覧スコア更新 API を非同期実行
- 折りたたみ状態は localStorage に保存
- 狭幅時は横スクロール fallback を適用

#### 非対応（曲詳細）

- 共有再生リスト
- 共同注釈

### 3.3 編集ページ（`/edit.html`）

#### 表示要素（編集）

- `title`, `slug`, `artist`, `tags`, `youtube`, `chordPro`
- 保存/削除アクション

#### 挙動（編集）

- `mode=add|edit` で動作分岐
- `youtube` は複数入力から `{id,start}` に正規化
- 削除は edit モード時のみ有効

## 4. API 基本設計

### 4.1 採用 API

- `GET /api/songs`
- `GET /api/songs-ranking`
- `GET /api/songs-search`
- `GET /api/song`
- `POST /api/songs-view/{id?}`
- `POST|PUT|DELETE /api/edit-song`
- `GET /api/youtube/search-duration`（参考時間推定）

### 4.2 非採用 API

- `/api/setlists*` 系エンドポイント（新規追加しない）

### 4.3 共通制御

- 全APIで認証済みアクセスを前提とする
- リクエスト principal から `ownerId` を解決する
- 取得/更新時に `ownerId` 条件を必須化する
- 認可違反は `403`、対象なしは `404` を返す

## 5. データ基本設計

### 5.1 Songs コンテナ

### パーティション

- Partition Key は `/ownerId`

### 主なフィールド

- `id`, `ownerId`, `slug`, `title`, `artist`, `tags[]`, `youtube[]`, `chordPro`
- `score`, `display_score`, `last_viewed_at`, `createdAt`, `updatedAt`

### 制約

- `id` は更新不可
- `youtube[].start` は 0 以上の整数
- 不正/空データは保存前に除外

### 5.2 localStorage

### 保存対象

- 表示設定、移調/表記、オートスクロール位置、注釈、手書き

### 方針

- 既存キー互換を維持
- キー名称変更時は移行処理を実装
- サーバー同期は実施しない

## 6. 認証・認可基本設計

### 6.1 認証

- `MyChordpro`: GitHub 認証を継続
- `chordwiki_personal`: MS 認証（参考情報）
- 認証方式は統合しない

### 6.2 認可

- ルート制御で `/api/*`, `/song*`, `/edit*` を認証必須にする
- API 層で ownerId 境界を再検証する
- UI の editor 表示は `/.auth/me` を元に切替する

### 6.3 共有仕様

- URL 共有は「画面遷移情報の共有」に留まる
- データ共有はしない
- 他人が同URLにアクセスしても他人の ownerId 空間が解決される

## 7. 移行設計

### 7.1 移行単位

- Step 1: UI 挙動同期（折りたたみ、補助パネル、レスポンシブ）
- Step 2: API 契約整理（不要分岐削減、入力正規化統一）
- Step 3: localStorage 互換整理（キー移行、既存データ保護）
- Step 4: 受入試験（認証境界、ownerId 境界、共有不可仕様）

### 7.2 後方互換

- 既存 URL と主要クエリは互換維持
- 既存 Song ドキュメントを再投入不要とする
- 既存 localStorage データを可能な限り保持する

## 8. テスト観点

### 8.1 機能試験

- ランキング/検索/詳細/編集/削除の正常系
- タグ検索とサジェスト
- YouTube 再生と推定時間取得

### 8.2 境界試験

- ownerId 不一致アクセスの拒否
- 未認証アクセスの遮断
- URL 共有時に他ユーザーのデータが見えないこと

### 8.3 非機能試験

- API 失敗時の画面劣化耐性
- モバイル狭幅での描画崩れ有無
- localStorage 容量超過時の安全動作

## 9. 受入条件

- セットリスト共有機能が存在しないこと
- GitHub 認証が維持されること
- ownerId 分離が維持されること
- URL を第三者へ共有してもデータ共有にならないこと
- 既存主要機能（閲覧、検索、編集、練習補助）が回帰しないこと
